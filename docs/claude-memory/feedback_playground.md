The playground at `/playground` is the gallery for every component, viz, widget, screen, and flow. Three rules govern how items are added and presented.

## 1. PlaygroundCard wrapper + visual hierarchy

Every item uses the shared `PlaygroundCard` component. Status tags and variant controls are visually distinct so lifecycle state never gets confused with content variant.

- **Status tag:** colored DlsTag pill, top-right of card header (read-only; see rule 2 below).
- **Variant switcher:** monochrome segmented control on its own row below the header.
- **Viz / Widgets:** 360px white content box with 24px padding inside the card. No device frame.
- **Screens / Flows:** `DeviceFrame` bezel wrapper inside the card.

**Why:** earlier playground designs mixed status pills and variant pills in the same row with identical styling — impossible to tell lifecycle state from content variant. The user also needed to see exact component footprint at device width.

**How to apply:** always use `PlaygroundCard` for new playground items. Never mix status and variant controls visually. Only use device frames for full-screen layouts (screens, flows).

## 2. Item lifecycle (status chips are auto-derived)

Status chips are **read-only and derived** from the codebase. Never hand-toggle. Three states:

- **Exploring** — local WIP. Default. The item exists only in the playground.
- **Confirmed** — design-lint has passed for this item and no source file has been edited since. Recorded by Claude in `app/preview/_shared/lint-passes.json` when the user runs design-lint on the item.
- **Integrated** — real app code (outside `app/(main)/playground` and `app/preview`) imports the item's source symbol, or constructs the relevant `ChatCard` type. Detected by a static scan.

**Resolution precedence:** integrated → confirmed → exploring. Code wins over lint records.

**How statuses change (no toggles):**
- Item gets imported into a real screen/flow → chip flips to `integrated` on next dev server start.
- User asks Claude to lint an item, it passes → Claude appends to `lint-passes.json` → chip flips to `confirmed`.
- Source file is edited after a lint pass → mtime check invalidates the pass → chip drops back to `exploring`.

**Why:** hand-toggled chips drifted constantly. Of 25 hand-set statuses, 12 were lying about real state when this was rebuilt. Statuses must come from the actual code, not memory.

**How to apply:**
- Never edit `app/preview/_shared/status-generated.ts` (auto-generated by `scripts/scan-playground-status.mjs`).
- Never reintroduce a click handler on the chip.
- When adding a new playground item, also add an entry to `app/preview/_shared/integration-manifest.json` so the scan knows what symbol / cardType to look for.
- When the user says "run design-lint" on an item and it passes cleanly, follow the "Recording a pass" section in `.claude/skills/design-lint/SKILL.md`.
- The scan runs automatically via `predev` and `prebuild` hooks in `package.json`. To run it manually: `pnpm scan-playground-status`.

**Discarded items:** no longer a status. To remove an item: commit it to git first, then delete from code and from `integration-manifest.json`.

## 3. States are controls, not variants

When wiring a component into `/playground/components`:

- A **variant** is a structurally distinct version of the component — different layout, different typography, different size, something that isn't reachable by toggling a prop. Example: OTP 4-digit vs 6-digit (different cell sizes per Figma). Example: AppChrome standard vs degen.
- A **state** is any prop the component already exposes: `disabled`, `status="error"`, `helperText`, `leading`, `selected`, etc. States are NEVER separate variant chips.

**Why:** adding a chip per state explodes the playground (a 5-prop component would mean dozens of chips), buries the actual variants, and makes it impossible to combine states (e.g. "error AND disabled"). The user has called this out as a recurring mistake to stop making.

**How to apply:**
1. Identify true variants (almost always 1 — sometimes 2-3 like OTP length).
2. For everything else use `useControlPanel` + `useSlotControls(panel)` from `app/preview/_shared/ControlPanel.tsx`. Reference implementation: `SnackbarPlayground` in `app/(main)/playground/components/page.tsx`.
3. Default playground variant name when there's only one: `"playground"`.

If you find yourself writing more than 3 variant chips for a single component, stop and ask whether they're real variants or states.
